# 架构设计

由于 [Ozon](https://docs.ozon.ru/api/seller/zh/?__rr=3#tag/Getting-started)现行的文档规定，API 的请求只允许在后台操作，backend 本该只需要负责和 Ozon 的 API 通信即可，数据是不需要本地存储的。但是 Ozon 官方的 API 有诸多限制，所以设计了 backend 进行本地数据的备份。

## 数据通信流

数据流通信有两条主线

1. **backend <---> Ozon API**
   1. 主动拉取
   2. 定时任务
2. **前端 <---> backend**
   1. 强制刷新
      1. 前端 ---> backend ---> ozon API
   2. 缓存模式
      1. 前端 ---> backend

backend 通过定时任务，主动拉取 ozon 的数据，存入数据库缓存。

### 二、定时任务系统

#### 2.1 架构

使用 APScheduler 作为任务调度引擎，所有任务配置存储在数据库 `task_configs` 表中，支持运行时动态增删改。

```
┌─────────────────────────────────────────────┐
│            APScheduler                       │
│                                              │
│  ┌──────────────┐  ┌──────────────────────┐  │
│  │ AsyncIOScheduler│  │   JobStore (DB)    │  │
│  │ (调度器)       │  │  持久化任务状态     │  │
│  └──────┬───────┘  └──────────────────────┘  │
│         │                                     │
│  ┌──────▼──────────────────────────────────┐  │
│  │          Job 工厂                         │  │
│  │  sync_orders  sync_finance  sync_stores  │  │
│  └─────────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

#### 2.2 任务类型

| 任务 key          | 说明             | 默认间隔  | Ozon API                              |
|-------------------|------------------|-----------|---------------------------------------|
| `sync_orders`     | 同步订单数据     | 30 分钟   | `/v2/posting/fbs/list`               |
| `sync_finance`    | 同步店铺资金     | 60 分钟   | `/v1/finance/statement`              |
| `sync_warehouses` | 同步仓库信息     | 24 小时   | `/v2/warehouse/list`                 |

#### 2.3 任务配置字段

| 字段              | 类型    | 说明                                 |
|-------------------|---------|--------------------------------------|
| `id`              | int     | 主键                                 |
| `task_key`        | string  | 唯一标识 (sync_orders / sync_finance) |
| `name`            | string  | 中文名称                             |
| `description`     | string  | 描述                                 |
| `trigger_type`    | string  | `interval` 或 `cron`                 |
| `interval_seconds`| int     | 间隔秒数 (trigger_type=interval)     |
| `cron_expression` | string  | Cron 表达式 (trigger_type=cron)      |
| `enabled`         | bool    | 是否启用                             |
| `last_run_at`     | datetime | 最后执行时间                        |
| `last_status`     | string  | 上次执行状态 (success/failed)        |

#### 2.4 店铺同步配置

每个店铺可以独立控制是否参与定时同步，通过 `store_sync_configs` 表存储：

| 字段                | 类型   | 说明                    |
|---------------------|--------|-------------------------|
| `store_id`          | int    | 关联店铺 ID             |
| `sync_orders`       | bool   | 是否同步订单            |
| `sync_finance`      | bool   | 是否同步流水            |
| `sync_warehouses`   | bool   | 是否同步仓库            |

### 三、Ozon API 客户端

`OzonClient` 封装与 Ozon Seller API 的所有通信，每个店铺实例化一个客户端。

```
OzonClient(client_id, api_key)
  ├── get_seller_info()        → 卖家信息
  ├── get_warehouses()         → 仓库列表
  ├── get_orders(filters)      → 订单列表 (FBS)
  ├── get_finance_statement()  → 资金报表
  └── _request(method, path)   → 底层 HTTP 请求
```

### 四、同步服务

每个同步类型对应一个 Service 函数，被 Scheduler 和手动刷新共用：

- `sync_orders_for_store(db, store)` — 拉取该店铺最新订单写入本地
- `sync_finance_for_store(db, store)` — 拉取资金数据
- `sync_warehouses_for_store(db, store)` — 同步仓库

### 五、配置管理

所有定时任务配置通过 REST API 管理：

| 方法     | 路径                                  | 说明                 |
|----------|---------------------------------------|----------------------|
| `GET`    | `/api/task-configs/`                  | 获取所有任务配置     |
| `PUT`    | `/api/task-configs/{key}`             | 修改任务配置         |
| `POST`   | `/api/task-configs/{key}/trigger`     | 手动触发一次任务     |
| `GET`    | `/api/store-sync-configs/`            | 获取店铺同步配置     |
| `PUT`    | `/api/store-sync-configs/{store_id}`  | 修改店铺同步配置     |

### 六、启动流程

```
FastAPI startup
    │
    ├── 数据库表创建 (Base.metadata.create_all)
    │
    ├── 初始化默认任务配置 (如数据库为空)
    │   ├── sync_orders     interval 1800s
    │   ├── sync_finance    interval 3600s
    │   └── sync_warehouses interval 86400s
    │
    └── 启动 APScheduler
        ├── 从 DB 加载所有 enabled 任务
        ├── 注册到调度器
        └── 开始调度
```
