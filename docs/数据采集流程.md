# 数据采集流程

> 本文档详细描述 auto-ozon 项目中,从数据采集到后端存储再到前端展示的完整链路。
> **所有数据采集统一通过浏览器插件入口**,采集数据直接保存到后端。
> 列表页采集时,插件会自动通过 Ozon 内部 JSON API 补全每个商品的品牌/分类/属性/描述等详细信息。

---

> **设计决策**:后端 HTTP Scraper (httpx) 和内部 API 抓取脚本已废弃。
> 原因:Ozon 有强反爬机制,HTTP 请求经常返回 403/验证码,且内部 API 需要浏览器 session cookies。
> 浏览器插件运行在真实浏览器环境中,拥有用户的 cookies 和 session,是最可靠的数据采集路径。
> 因此所有采集入口统一收敛到浏览器插件。

## 二、浏览器插件采集

### 2.1 技术栈

| 组件 | 技术 | 文件位置 |
|------|------|----------|
| 插件框架 | WXT (Web Extension Tools) | `browser-extension/` |
| UI 组件 | Vue 3 + Tailwind CSS | `browser-extension/src/components/` |
| Content Script | WXT content script API | `browser-extension/src/entrypoints/ozon.content.ts` |
| Background | WXT background service worker | `browser-extension/src/entrypoints/background.ts` |
| 后端通信 | Fetch API | `browser-extension/src/utils/api.ts` |

### 2.2 两种采集模式

#### 模式 A: 商品详情页采集 (单个商品)

**触发条件**:URL 匹配 `/product/xxx-123456/` (路径末尾包含数字 ID)

**流程**:
1. 用户在 Ozon 商品详情页点击插件 "一键采集" 按钮
2. Content Script (`ozon.content.ts`) 中的 `scrapeOzonProduct()` 从 DOM 提取完整商品信息
3. 数据通过 Background Script (`background.ts`) 直接 POST 到后端 `POST /api/browser-sync/sync-products`
4. 后端去重后写入 `scraped_product_records` 表

**提取字段**:

| 字段 | 提取方式 | DOM 选择器 / 逻辑 |
|------|----------|-------------------|
| `sourceId` | URL 路径正则 | `location.pathname.match(/\/(\d+)\/?$/)` |
| `images` | 图片列表 | `[data-widget="webGallery"] img`, 替换 `/wc\d+/` → `/wc500/` 获取高清 |
| `price` | 文本解析 | `[data-widget="webPrice"] .tsHeadline600Large` |
| `oldPrice` | 删除线价格 | `[data-widget="webPrice"] span.pdp_bj` 或 `line-through` 样式的 span |
| `title` | 标题 | `h1` 或 `[data-widget="webProductHeading"] span` |
| `rating` | 评分 | `[data-widget="webReviewProductScore"]` 中的数字 |
| `reviewCount` | 评论数 | 从评分附近文本提取整数 |
| `brand` | 品牌 | `[data-widget="webBrandName"] a` 或 `a[href*="/brand/"]` |
| `category` | 分类 | `[data-widget="webBreadcrumb"] a` 面包屑导航 |
| `attributes` | 属性列表 | `[data-widget="webCharacteristics"]` 内的所有属性名/值对 |
| `description` | 描述 | `[data-widget="webDescription"]` 文本 |
| `sellerName` | 卖家名 | `[data-widget="webMerchantInfo"]` 或 `[data-widget="webBestSeller"]` |

#### 模式 B: 列表页滚动采集 + 自动补全 (批量商品)

**触发条件**:URL 匹配 `/category/...`、`/brand/...`、`/search/...`、`/seller/...`、`/collection/...`

**流程**:
1. 用户在 Ozon 分类/搜索/品牌列表页点击插件 "滚动采集全部商品" 按钮
2. Content Script 中的 `scrollAndCollect()` 启动自动滚动
3. 每次滚动一屏后,`scanListCards()` 扫描页面上所有商品卡片
4. 连续 3 次无新增商品时停止,或达到 maxItems 上限
5. **自动补全阶段**:逐个调用 Ozon 内部 JSON API (`fetchProductDetailFromApi`) 获取每个商品的品牌/分类/属性/描述/图片等完整详情
6. 补全后的完整商品数据批量 POST 到后端

> **重要变更**:列表页采集不再只收集摘要,而是在采集卡片后立即自动补全每个商品的完整详情,无需用户手动触发补全。
> 每个商品之间间隔 1.5~3 秒随机延时,避免触发 Ozon 风控。

**商品卡片提取逻辑**:

```
扫描选择器优先级:
1. [class*="tile-root"]                            → Ozon 实际卡片类
2. [data-widget="tileGridDesktop"] > div           → 标准网格
3. [data-widget="searchResults"] > div             → 搜索结果
4. [class*="widget-search-result"] > div           → 搜索结果备选
5. [data-widget*="tile"] > div                     → 任意 tile widget
6. [class*="tile-hover"]                           → tile-hover 卡片类
7. a[href*="/product/"] 最近的 div 容器            → 兜底方案
```

**每个卡片提取字段**:

| 字段 | 提取方式 |
|------|----------|
| `sourceId` | 从 `a[href*="/product/"]` 的 href 中正则提取 |
| `title` | `span[class*="tsBody"]` 或 `a span` |
| `price` | 所有含 `₽` 的 span 中,非删除线的最大值 |
| `oldPrice` | 含 `₽` 且 `line-through` 的 span |
| `imageUrl` | 卡片内 `img.src` (替换为高清 URL) |
| `rating` | `[class*="rating"]` 或 `[class*="star"]` 中的数字 |
| `reviewCount` | `[class*="review"]` 或 `[class*="comment"]` 中的整数 |
| `sourceUrl` | 拼接为完整 URL |

**滚动采集参数**:
- `maxItems`:最大采集数量 (默认 200)
- `scrollDelay`:每次滚动间隔 (默认 1500ms)
- `MAX_STABLE`:连续无新增轮数阈值 (默认 3 次)
- `maxPages`:最大翻页数 (默认 10,安全上限)

**翻页支持**:滚动采集支持自动翻页。当当前页滚动到底部后,会自动查找并点击"下一页"按钮,等待新页面加载后继续滚动采集。翻页策略:
1. `[data-widget="paginator"]` 内的 `rel="next"` 或 "Далее" 按钮
2. 全局 `rel="next"` 链接
3. URL `page=N` 参数递增链接
4. 分页导航中的 ">" / ">>" 按钮

### 2.3 插件 Popup UI

| 页面 | 显示内容 | 按钮行为 |
|------|----------|----------|
| 商品详情页 | "商品页" 标签 + "一键采集商品" | 触发 `triggerScrape` → 后端 |
| 列表页 | "列表页" 标签 + "滚动采集全部商品" | 触发 `triggerListScrape` → 滚动 + 批量后端 |
| 其他页面 | "不支持采集" 提示 | 无按钮 |
| 不支持平台 | "当前页面不支持采集" | 无按钮 |

### 2.4 Background Script 消息路由

**文件**: `browser-extension/src/entrypoints/background.ts`

| 消息 action | 来源 | 处理逻辑 |
|-------------|------|----------|
| `checkCurrentPage` | Popup | 查询当前 tab 的页面类型(product/list/unknown) |
| `triggerScrape` | Popup | 向 content script 发 `scrape` → 单个商品采集,采集完成后直接保存后端 |
| `triggerListScrape` | Popup | 向 content script 发 `scrapeList` → 滚动采集 + 自动补全 → 批量保存后端 |
| `productScraped` | Content | 接收单个商品 → 直接保存后端 |
| `getProducts` | RecordsPanel | 从后端查询产品列表 |
| `deleteProduct` | RecordsPanel | 从后端删除产品 |
| `listProgress` | Content | 列表采集进度 (转发给 popup) |
| `enrichProgress` | Content | 自动补全进度 (转发给 popup) |

### 2.5 数据流向

```
模式 A (商品详情页):
  用户点击"一键采集"
  → Content Script scrapeOzonProduct() 解析 DOM (完整字段)
  → Background triggerScrapeInTab() → handleProductScraped()
  → POST /api/browser-sync/sync-products
  → SQLite → 前端展示

模式 B (列表页):
  用户点击"滚动采集"
  → Content Script scrollAndCollect() 自动滚动+翻页+扫描卡片
  → 每个卡片只提取摘要 (标题/价格/评分/sourceId)
  → 逐个调用 fetchProductDetailFromApi(sourceId) 自动补全
     (通过 Ozon 内部 JSON API 获取品牌/分类/属性/描述/图片等完整详情)
     (每个请求间隔 1.5~3s 随机延时)
  → 补全后的完整 ScrapedProduct[] 批量返回 Background
  → Background triggerListScrapeInTab() → POST /api/browser-sync/sync-products
  → SQLite → 前端展示
```

## 三、废弃路径 (已整合到浏览器插件)

> 以下路径因 Ozon 反爬限制已废弃,功能已整合到浏览器插件中。

### 3.1 后端 HTTP Scraper (已废弃)

**原文件**: `backend/app/services/scrapers/ozon.py`

使用 `httpx` 直接请求 Ozon 页面 HTML,解析提取数据。**问题**:Ozon 有强反爬机制,直接 HTTP 请求经常返回 403 或验证码,成功率极低。

### 3.2 内部 API 抓取脚本 (已废弃)

**原文件**: `backend/app/services/scrapers/ozon_product_scraper.py`

利用 Ozon 内部 `entrypoint-api.bx` JSON API 批量抓取。**问题**:需要浏览器 session cookies,纯 Python httpx 请求被 403 拒绝。

### 3.3 为什么统一到浏览器插件

| 对比项 | 后端 HTTP Scraper | 内部 API 脚本(后端) | 浏览器插件(DOM) | 浏览器插件(内部 JSON API) |
|--------|-------------------|---------------------|-----------------|--------------------------|
| 反爬对抗 | 经常 403 | 需要 cookies,403 | 无限制 | 无限制(携带浏览器 cookies) |
| 数据完整度 | 有限属性 | 5-10 个属性 | 完整 (详情页) | 完整 (widget 结构化数据) |
| 运行环境 | 需要代理 | 需要浏览器 session | 真实浏览器 | 真实浏览器 |
| 适用场景 | 废弃 | 废弃 | 模式 A (单个商品) | 模式 B 补全 (批量商品) |
| 用户操作 | 无 | 手动运行脚本 | 点击按钮 | 自动(采集时触发) |

## 五、Ozon 内部 API 自动补全

### 5.1 概述

列表页采集时,Content Script 会自动通过 Ozon 内部 JSON API 补全每个商品的完整详情。该 API 运行在浏览器环境中,可以携带用户的 cookies,因此不受反爬限制。

**文件**: `browser-extension/src/entrypoints/ozon.content.ts`

### 5.2 API 地址

```
https://www.ozon.ru/api/entrypoint-api.bx/page/json/v2?url=/product/-{sourceId}/
```

### 5.3 请求方式

```typescript
fetch(url, {
  headers: {
    'accept': 'application/json',
    'x-requested-with': 'XMLHttpRequest',
  },
  credentials: 'include', // 携带用户 cookies
})
```

### 5.4 解析逻辑 (`parseInternalApiResponse`)

返回的 JSON 结构中 `widgetStates` 包含所有 widget 数据,按 key 匹配提取:

| widget key 匹配 | 提取字段 | 提取逻辑 |
|------------------|----------|----------|
| `webProductHeading` / `pdptitle` | `title` | `widget.title \|\| widget.text` |
| `webGallery` / `gallery` | `images` | `widget.images[].big \|\| medium \|\| small` |
| `webPrice` / `price` | `price`, `oldPrice` | 解析 `widget.price` / `widget.oldPrice`,去除空格和逗号 |
| `webReviewProductScore` / `review` | `rating`, `reviewCount` | `widget.rating` / `widget.reviewCount` |
| `webBrandName` / `brand` | `brand` | `widget.brand?.name \|\| widget.text` |
| `webCharacteristics` / `characteristic` | `attributes[]` | `widget.options[].title + value` |
| `webDescription` / `description` | `description` | `widget.text \|\| widget.content` |
| `webMerchantInfo` / `webBestSeller` / `merchant` | `sellerName` | `widget.merchantName \|\| widget.text` |
| `webBreadcrumb` / `breadcrumb` | `category` | 面包屑数组,跳过第一项后用 `>` 连接 |

### 5.5 补全时机

1. **列表页自动补全**:模式 B 采集完成后,逐个调用 `fetchProductDetailFromApi()` 补全,每个请求间隔 1.5~3 秒随机延时
2. **旧版手动补全**(已移除):之前通过前端"补全详情"按钮触发,现改为采集时自动完成

### 5.6 数据合并策略

补全后的字段采用"有值优先"策略覆盖摘要数据:

```typescript
products[i] = {
  ...p,  // 卡片摘要作为基础
  title: detail.title || p.title,
  brand: detail.brand || p.brand,
  category: detail.category || p.category,
  images: detail.images?.length > 0 ? detail.images : p.images,
  attributes: detail.attributes?.length > 0 ? detail.attributes : p.attributes,
  price: detail.price || p.price,
  // ...
}
```

### 6.3 数据转换规则

插件使用**驼峰命名**,后端使用**蛇形命名**,`browser_sync.py` 中做了映射:

```
sourceId     → source_id
oldPrice     → old_price
reviewCount  → review_count
sellerUrl    → seller_url
sourceUrl    → source_url
scrapedAt    → scraped_at
```

#### 操作功能

- **编辑**:弹出 Modal 编辑商品的标题/品牌/品类/价格/描述/链接 → `PUT /api/selection/products/{id}`
- **上传**:弹出 Modal 选择目标店铺和 Offer ID → `POST /api/selection/products/{id}/upload` → 调用 Ozon API 创建商品
- **删除**:Popconfirm 确认后 → `DELETE /api/selection/products/{id}`

## 八、数据流向总结

### 模式 A:商品详情页采集

```
用户在 Ozon 商品页点击"一键采集"
    ↓
Content Script: scrapeOzonProduct()
    → DOM 解析完整商品信息 (标题/价格/属性/描述/图片/品牌/卖家等)
    → 提取 sourceId (URL 路径正则)
    ↓
Background Script: triggerScrapeInTab() → handleProductScraped()
    → POST /api/browser-sync/sync-products
    ↓
browser_sync.py → bulk_create_scraped_products() (去重写入)
    ↓
SQLite: scraped_product_records 表
    ↓
前端 GET /api/selection/products → ProductSelectionView.vue 展示
```

### 模式 B:列表页滚动采集 + 自动补全

```
用户在 Ozon 分类/搜索/品牌页点击"滚动采集全部商品"
    ↓
Content Script: scrollAndCollect()
    → 自动滚动页面 (scrollDelay=1500ms)
    → scanListCards() 扫描所有商品卡片
    → 连续 3 轮无新商品时停止
    ↓
自动翻页: findAndClickNextPage()
    → 检测分页导航 (paginator/rel=next/page=N)
    → 点击下一页,等待新内容加载
    → 继续滚动采集 (最多 maxPages=10 页)
    ↓
卡片摘要提取 (仅标题/价格/评分/sourceId/imageUrl)
    ↓
自动补全阶段 (逐个调用 Ozon 内部 JSON API):
    fetchProductDetailFromApi(sourceId)
    → GET https://www.ozon.ru/api/entrypoint-api.bx/page/json/v2?url=/product/-{id}/
    → parseInternalApiResponse() 解析 widget 数据
    → 获取: title, brand, category, images, attributes, description, sellerName, price, rating...
    → 每个请求间隔 1.5~3 秒随机延时 (避免风控)
    ↓
补全后的完整 ScrapedProduct[] 批量返回 Background
    ↓
Background Script: triggerListScrapeInTab()
    → POST /api/browser-sync/sync-products (批量)
    ↓
browser_sync.py → bulk_create_scraped_products() (批量去重写入)
    ↓
SQLite: scraped_product_records 表
    ↓
前端 GET /api/selection/products → ProductSelectionView.vue 展示
```

## 九、关键限制与注意事项

### 9.1 Ozon 反爬限制

- **直接 HTTP 请求**(`ozon.py`):经常返回 403 / 307 重定向 / 验证码,成功率低
- **后端内部 API 脚本**(`ozon_product_scraper.py`):需要浏览器 session cookies,纯 Python httpx 请求被 403 拒绝
- **浏览器插件 DOM 解析**(`ozon.content.ts`):在真实浏览器环境中运行,不受反爬限制,是最可靠的数据采集路径
- **浏览器插件内部 JSON API**(`fetchProductDetailFromApi`):利用 `credentials: 'include'` 携带浏览器 cookies 请求 Ozon 内部 API,同样不受反爬限制

### 9.2 数据去重

- 数据库层面:`(platform, source_id)` 唯一约束,bulk insert 时按此去重
- 两个来源可能产生重复:插件采集同一商品 → 浏览器插件同步到后端 + 手动通过 JSON 导入 → 数据库会跳过重复

### 9.3 两种采集模式的数据质量对比

| 对比项 | 模式 A (商品详情页) | 模式 B (列表页 + 自动补全) |
|--------|---------------------|--------------------------|
| 数据来源 | 单次 DOM 解析 | 卡片摘要 + 内部 JSON API 补全 |
| 价格准确度 | DOM 实时解析 | DOM 摘要 + API 补全 |
| 属性完整度 | 完整 (DOM 提取) | 完整 (内部 API 提取) |
| 品牌/分类 | DOM 提取 | 内部 API 提取 |
| 图片质量 | 高清 URL (wc500 替换) | API 返回 big URL |
| 卖家信息 | DOM 提取 | 内部 API 提取 |
| 采集速度 | 快 (单次点击) | 慢 (滚动 + 翻页 + 逐个补全,每商品 1.5~3s) |
| 适用场景 | 精准采集特定商品 | 批量采集分类/搜索结果 |

> 注意:`ozon_smartphones.json` 中的 price 是字符串如 `"22 562 ₽"`,导入时需要解析数字。
> `selection.py` 的 `_parse_price()` 函数已实现解析逻辑,去除空格和 ₽ 符号后转为浮点数。

### 9.4 内部 JSON API 限制

- API 地址格式:`/api/entrypoint-api.bx/page/json/v2?url=/product/-{sourceId}/`
- 需要浏览器环境中的 cookies,直接 Python 请求会被 403
- 响应中的 `widgetStates` 结构可能随 Ozon 前端更新而变化,需要定期检查
- 部分商品可能缺少某些 widget 数据 (如新上架商品可能没有评价)

### 9.5 后端服务端口

| 服务 | 端口 | 启动方式 |
|------|------|----------|
| Backend (FastAPI) | 9000 | `uvicorn app.main:app --port 9000` |
| Frontend (Vite) | 5173/5174 | `npm run dev` |
