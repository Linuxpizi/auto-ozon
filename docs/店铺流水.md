# 店铺流水

## 概述

在 Ozon 现行的严苛制度下，时刻监视店铺的资金流动是非常重要的。在遇到非常规难以理解的扣款项时，可以及时发送邮件或者企业微信、飞书等通知。

本模块负责：**余额概览、财务明细、数据同步、异常告警**。

---

## 涉及的 Ozon API

### 1. 获取余额报告 `POST /v1/finance/balance`

对应卖家个人中心 → 财务 → 余额 模块。报告期最长间隔 **30 天**。

**请求**
```json
{
  "date_from": "2025-06-01",
  "date_to": "2025-06-30"
}
```

**响应关键字段**
| 路径 | 说明 |
|------|------|
| `total.opening_balance.value` | 期初余额 |
| `total.closing_balance.value` | 期末余额（= 当前可用余额） |
| `total.accrued.value` | 待结算金额 |
| `total.payments[].value` | 已打款金额（可累加） |
| `cashflows.sales.amount.value` | 销售收入（订单金额） |
| `cashflows.sales.fee.value` | 销售佣金/手续费 |
| `cashflows.returns.amount.value` | 退货金额 |
| `cashflows.returns.fee.value` | 退货手续费 |
| `cashflows.services[].amount.value` | 各项服务费（可累加） |

> 本地实现位于 `ozon_client.py::get_balance_report()`，将上述字段映射为 `closing_balance`、`sales_amount`、`sales_fee`、`returns_amount`、`services_cost`、`accrued`、`paid` 返回。

### 2. 获取财务报告 `POST /v1/finance/cash-flow-statement/list`

从1号到15号以及从16号到31号的财务报告。请求一天的报告时，会返回 15 天的报告。

**请求**
```json
{
  "date": {
    "from": "2025-06-01T00:00:00.000Z",
    "to": "2025-06-30T23:59:59.999Z"
  },
  "with_details": true,
  "page": 1,
  "page_size": 100
}
```

**响应关键字段**
| 路径 | 说明 |
|------|------|
| `result.cash_flows[].orders_amount` | 订单金额 |
| `result.cash_flows[].returns_amount` | 退货金额（负数） |
| `result.cash_flows[].commission_amount` | 佣金 |
| `result.cash_flows[].services_amount` | 服务费 |
| `result.cash_flows[].item_delivery_and_return_amount` | 配送与退货费用 |
| `result.cash_flows[].currency_code` | 币种 |
| `result.cash_flows[].period` | 报告周期 (begin/end/id) |
| `result.details.begin_balance_amount` | 期初余额（with_details=true 时） |
| `result.details.end_balance_amount` | 期末余额（with_details=true 时） |
| `result.details.services[].name/price` | 服务费明细 |
| `result.details.others[].name/price` | 其他费用明细 |
| `result.details.delivery/return.total` | 配送/退货服务费总计 |
| `page_count` | 总页数 |

> 本地实现位于 `ozon_client.py::get_finance_statement()`，默认 `with_details=False` 仅拉取汇总数据。

## 数据模型

### DB 表 `store_finances` (每店一行)

| 列 | 类型 | 说明 |
|----|------|------|
| `store_id` | INTEGER UNIQUE FK | 关联 stores 表，ON DELETE CASCADE |
| `store_name` | VARCHAR(128) | 店铺名称 |
| `opening_balance` | FLOAT | 期初余额 |
| `balance` | FLOAT | 可用余额 (closing_balance) |
| `total_income` | FLOAT | 总收入 (sales_amount) |
| `total_expense` | FLOAT | 总支出 (sales_fee + services_cost) |
| `paid` | FLOAT | 已打款金额 |
| `pending_amount` | FLOAT | 待结算金额 (accrued) |
| `last_sync_at` | DATETIME | 最后同步时间 |

### DB 表 `finance_cash_flows` (每半月周期一行)

| 列 | 类型 | 说明 |
|----|------|------|
| `store_id` | INTEGER FK | 关联 stores 表 |
| `store_name` | VARCHAR(128) | 店铺名称 |
| `period_id` | INTEGER | Ozon 周期 ID |
| `period_begin/end` | VARCHAR(64) | 周期起止时间 |
| `orders_amount` | FLOAT | 订单金额 |
| `returns_amount` | FLOAT | 退货金额（负数） |
| `commission_amount` | FLOAT | 佣金 |
| `services_amount` | FLOAT | 服务费 |
| `delivery_amount` | FLOAT | 配送与退货费用 |
| `currency_code` | VARCHAR(16) | 币种，默认 RUB |

### Pydantic Schema

| 类 | 用途 |
|----|------|
| `StoreFinanceUpdate` | 部分更新（balance/paid 等 Optional） |
| `StoreFinanceRead` | 返回前端（含 id + last_sync_at） |
| `FinanceCashFlowBase/Read` | 财务报告读写 |

## 同步流程

```
用户点击"一键同步" 或 定时任务触发
        │
        ▼
  POST /api/finances/sync
        │
        ▼
  sync_service.run_sync_task(db, "sync_finance")
        │
        ▼
  遍历所有 status=active 的店铺
        │
        ▼
  sync_finance_for_store(db, store)
        │
        ├─① 分段拉取余额报告 (每段≤30天，共90天)
        │     OzonClient.get_balance_report(date_from, date_to) × 3次
        │     → 取最后一次的 closing_balance 作为最新余额
        │
        ├─② 拉取财务报告明细
        │     OzonClient.get_finance_statement(date_from, date_to)
        │     → 按 period_id upsert 到 finance_cash_flows 表
        │
        └─③ 更新余额汇总
              finance_crud.upsert_finance(db, store.id, update)
              → 写入 store_finances 表 (含 paid 字段)
```

**30 天窗口分段**：余额报告 API 限制 `date_from` ~ `date_to` 最长 30 天。同步时从 `now-90天` 开始，每次 30 天为一个窗口，取最后一次窗口的 `closing_balance` 作为最新余额。

## 后端 API 路由

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/finances/cashflows` | 获取财务报告明细（可选 `?store_id=&limit=`） |
| GET | `/api/finances/` | 获取所有店铺资金记录（可选 `?store_id=` 过滤） |
| GET | `/api/finances/{id}` | 获取单条资金记录 |
| POST | `/api/finances/` | 创建资金记录 |
| PUT | `/api/finances/{id}` | 更新资金记录 |
| PUT | `/api/finances/store/{store_id}/sync` | 为指定店铺 upsert 资金数据 |
| POST | `/api/finances/sync` | 一键同步所有店铺资金（触发 sync_service） |
| DELETE | `/api/finances/{id}` | 删除资金记录 |

### 交互说明

1. **一键同步**：调用 `POST /api/finances/sync`，触发 30 天窗口分段拉取 + 财务明细持久化，完成后自动刷新
2. **刷新**：直接拉取本地 DB 已同步的数据（不调 Ozon API）
3. **余额着色**：正数绿色、负数红色
4. **财务报告**：展示 `finance_cash_flows` 表中的真实 Ozon 半月周期数据

## 余额报告 vs 财务报告对比

| 维度 | 余额报告 | 财务报告 |
|------|---------|---------|
| Ozon API | `/v1/finance/balance` | `/v1/finance/cash-flow-statement/list` |
| 粒度 | 店铺级汇总 | 半月周期级明细 |
| 用途 | 一眼看清店铺资金概况 | 分析每笔收支的来源和趋势 |
| 本地存储 | `store_finances` 表（每店一行） | **未持久化**，前端从余额数据模拟生成 |
| 数据来源 | period 最长 30 天 | 按半月周期分段，含 with_details 详细分类 |

## 当前实现的问题与优化方向

### 已知问题

1. **30 天窗口限制未处理**：`sync_finance_for_store` 一次拉 90 天数据，但 Ozon balance API 限制 `date_from` 与 `date_to` 间隔最长 30 天。超出部分可能返回不准确的 opening/closing。应按 30 天窗口分段拉取。

2. **财务报告未持久化**：当前 `get_finance_statement()` 的返回值被丢弃，前端 CashFlowItem 是从余额数据模拟生成的，不是真实的逐笔流水。

3. **缺少异常检测/告警**：当前只有手动同步，缺少扣款异常检测和通知推送。

### 优化方向

| 优先级 | 项目 | 说明 |
|--------|------|------|
| P0 | 30 天窗口分段拉取 | 确保余额报告数据准确 |
| P1 | 财务报告持久化 | 新建 `finance_details` 表存储每条 cash_flow 记录 |
| P1 | 财务报告分页 | 前端支持翻页，后端按 page/page_size 返回 |
| P2 | 扣款异常检测 | 服务费 / 佣金 / 退货费用超过阈值时自动告警 |
| P2 | 通知集成 | 企业微信 / 飞书 Webhook 推送异常告警 |
| P3 | 定时自动同步 | 接入 scheduler_service，每日自动拉取 |
| P3 | 币种处理 | 当前假设 RUB → CNY 汇率转换 |

---

## 总结

店铺流水模块已实现基本的**一键同步 + 余额报告展示**功能：

- **后端**：通过 OzonClient 调用 `/v1/finance/balance` 获取店铺余额数据，upsert 到 `store_finances` 表；`/v1/finance/cash-flow-statement/list` 的调用已实现但结果未持久化
- **前端**：双 tab 布局（余额报告 + 财务报告），余额报告从 DB 读取真实数据，财务报告当前为模拟数据
- **核心瓶颈**：余额报告的 30 天窗口限制需分段拉取才能保证准确性；财务报告需新增存储层才能展示真实明细

下一步应优先解决 **30 天窗口分段拉取** 和 **财务报告持久化**，这两个改进将直接影响数据的准确性和可用性。
