# 阶段性目标

> 当前阶段直接采集 Ozon 官方数据存在很大的技术性难题，所以现阶段采取折中的方式进行处理。
> 先利用 BCS 的官方插件进行关键性的数据采集，然后利用 Ozon 官方的 API 对数据进行上报处理。
>
> 现状是 BCS 插件，对产品列表进行了很多关键性的数据指标追加，我们需要设计的功能如下
> 1. 对 BCS 追加的数据进行采集
> 2. 对产品特征关键数据进行采集
> 3. 这一步是必须要的，一定需要对数据在 1688 进行找同款，目的是一定要保证出了订单，一定可以正常的发货

## 开发步骤

> 原则: 以下的步骤，每完成一步，必须要验证其正确性

### 当前阶段：采集链路已贯通

- [x] 采集 **BCS** 追加的商品卡片信息
  - 只读取商品卡片内带明确 `bcs` 标记且可见的追加节点，避免误采 Ozon 原生卡片文本。
  - 支持 BCS 节点晚于商品卡片异步注入；重复扫描时增量合并，不清空已采集事实。
- [x] 采集 SKU 的必要属性
  - [x] 颜色列表：只从明确颜色事实和真实 SKU 变体颜色维度汇总，不从标题猜测。
  - [x] 关于商品：统一保存为可追溯事实 `facts`。
  - [x] 描述：保留详情页 DOM、结构化数据及 Ozon 内部接口采集结果。
  - [x] 特征：保存事实及物理规格，并持久化真实 SKU 的完整变体组合。
  - [x] 逐 SKU 商业事实：每个真实 SKU 独立保存颜色/规格组合、价格、原价、库存、条码和图片；不把商品级价格复制为全部 SKU 价格。
- [x] 打通扩展上报到后端持久化
  - 扩展上报 `facts`、`colorList`、`variants`。
  - 后端完成 Pydantic camelCase 转换、SQLAlchemy JSON 字段、旧 SQLite 表启动迁移。
  - 重复同步按事实、颜色、SKU 增量合并；列表页空数据不会覆盖详情页数据。
  - 变体仅按精确 SKU 合并：同 SKU 的互补字段可补全、最新显式价格/库存可更新且缺失字段不清空，不同 SKU 不互相复制属性或商业数据。
- [x] 在选品中心展示采集数据
  - 编辑抽屉原始数据区只读展示 `color_list`、`facts` 和真实 `variants`，并与人工维护的 `sku_list` 分区。
  - 兼容旧记录、JSON 字符串、空字段和异常 JSON；空采集结果不显示冗余区块。
  - 保存标题、描述等人工编辑内容时同步保留采集字段，避免误清空已持久化数据。
  - 真实变体按 SKU 独立卡片展示颜色/规格、价格、原价、库存、条码、图片和采集来源，并继续与人工 `sku_list` 隔离。
